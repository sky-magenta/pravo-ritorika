# E2E-тесты pravo-ritorika

Набор сквозных (end-to-end) тестов: скилл загружается в системный контекст модели,
каждому примеру подаётся вход, а ответ модели сверяется с ожиданием. Проверяет то,
ради чего скилл существует, — **находит ли он реальный дефект подачи и не выдумывает ли
уязвимости** там, где позиция подана убедительно.

- **прогон №1** (53 примера): 48/53 → 52/52 после исправлений. Поймано: перепутанная разметка D9;
  хрупкость alias-грейдинга clean-вердиктов (добавлены сигналы + регэксп-фолбэк).
- **прогон №2** (полный, 119 примеров): 117/119 → **119/119** после исправлений. Поймано:
  внутреннее противоречие набора (N5 объявлял «заранее закрытую давность» корректной, тогда как V7
  верно считает упреждающее закрытие незаявленной давности подаренным возражением — N5 исправлен);
  растяжимость термина «слепота ко второму адресату» (в `rhetoric-errors.md` добавлено
  разграничение с задачей адаптации, вход A6 уточнён).

## Что внутри

| Файл | Назначение |
|---|---|
| `cases.jsonl` | размеченные примеры (генерируется из `build_cases.py`) |
| `build_cases.py` | источник истины: примеры как Python-данные → пишет `cases.jsonl` |
| `validate_cases.py` | офлайн-проверка структуры и покрытия (без сети, годится для CI/pytest) |
| `run_evals.py` | реальный прогон через Claude API + грейдинг |
| `results.json` | отчёт последнего прогона (создаётся `run_evals.py`) |

## Устройство примера

```json
{"id":"D1","mode":"structure","category":"D.похороненный_тезис","difficulty":"easy",
 "input":"Проверь структуру меморандума для клиента: сначала три страницы о рисках, потом вывод…",
 "expect":"defect","defect_ru":"Похороненный тезис","defect_lat":"",
 "aliases":["похороненный тезис","вывод в середине","managerial reading"],
 "why":"Для клиента вывод должен идти первым; здесь он спрятан за риски."}
```

- **mode** — `stress-test` / `status` / `structure` / `adapt`.
- **expect** — `defect` (должен назвать дефект подачи из `aliases`) или `clean` (не должен выдумать уязвимость).
- Все примеры **условны**: реальные нормы и дела не воспроизводятся (инвариант скилла). Скилл проверяет
  **убедительность**, а не валидность вывода и не право по существу.

## Покрытие

- **Режимы:** stress-test, status, structure, adapt (покрыты все четыре).
- **Разделы** (по первой букве id) — каталог дефектов подачи из `references/rhetoric-errors.md`:
  - `S` — линия позиции (статус): смешение статусов, перевёрнутая лестница, неготовность к чужому
    статусу (правило зеркала), терминологическая капитуляция;
  - `D` — структура (диспозиция): похороненный тезис, инверсия гомерова порядка, слабый на вылет,
    взаимный подрыв, оценка в narratio, новый аргумент в заключении, перегруженное начало;
  - `B` — баланс средств убеждения: пафос вместо логоса, логос без этоса, передержка в пересказе,
    тотальная полемика, категоричность сверх нужного;
  - `V` — работа с возражениями: глухота к контрпозиции, преждевременная уступка, подаренное
    возражение, чужая повестка;
  - `T` — стресс-тест: удар по несущему, инвентаризация молчания, честная контрпозиция;
  - `A` — адресация: чужой регистр, слепота ко второму адресату, тревога вместо информации,
    отсутствие пути к согласию;
  - `E` — недобросовестные приёмы оппонента (эристика Шопенгауэра + классификация Поварнина,
    инвертированно): расширение тезиса, диверсия, нагруженный ярлык, «чтение в сердцах»,
    «палочный довод» + честный контрпример (добросовестный довод оппонента — не уловка);
  - `M` — M&A / корпоративные сценарии: индемнити и заверения, due diligence-меморандум,
    опционы, сделки с заинтересованностью, крупные сделки, переговоры по цене — риторические
    дефекты на материале практики + 2 clean;
  - `N` — корректные примеры (подача работает): ловят выдуманные уязвимости.
- **`clean`-примеры** — осознанная статусная лестница, гомеров порядок и порядок проверки,
  managerial reading, честный пафос на фактуре, управляемая уступка, закрытое молчание, переговорный
  путь к согласию: ловят галлюцинации дефектов.

## Быстрый старт

Офлайн-проверка набора (без ключа и сети):

```bash
python evals/validate_cases.py      # отчёт по структуре и покрытию
pytest evals/validate_cases.py      # то же как тест
```

Пересобрать `cases.jsonl` после правки `build_cases.py`:

```bash
python evals/build_cases.py
```

Реальный прогон (нужен ключ). Экспортируйте `ANTHROPIC_API_KEY` или войдите `ant auth login`:

```bash
python evals/run_evals.py --dry-run           # проверить сборку промптов без сети
python evals/run_evals.py --limit 5           # быстрый прогон 5 примеров
python evals/run_evals.py                      # весь набор, грейдер alias, модель claude-opus-4-8
python evals/run_evals.py --grader llm         # грейдинг вызовом-судьёй (точнее для clean)
python evals/run_evals.py --mode stress-test   # только один режим
```

## Как считается «прошёл»

- **`--grader alias` (по умолчанию, быстрый и бесплатный):** для `defect` — ответ модели содержит
  любой из `aliases` / имя дефекта; для `clean` — есть вердикт «устойчива / подача работает / без
  дефектов подачи».
- **`--grader llm` (точнее):** отдельный вызов-судья сверяет ответ с ожиданием по существу.
  Рекомендуется для `clean`-примеров: имя дефекта может звучать по-разному, а факт «уязвимость не
  выдумана» надёжнее оценивает судья, чем поиск подстрок.

## Стоимость и кэширование

Системный контекст (весь скилл) **одинаков для всех запросов** и кэшируется prompt caching'ом —
платится по сути один раз, дальше по цене чтения кэша. Модель по умолчанию — `claude-opus-4-8`,
adaptive thinking + `effort=high` (при старом SDK параметры автоматически опускаются).

## Замечание

Это оценка ФОРМЫ разбора: назван ли нужный дефект подачи / не выдумана ли лишняя уязвимость.
Юридическую верность самих формулировок (нормы, практику) скилл не устанавливает и оценка не
проверяет — такие места скилл помечает флагом `[требует проверки]`, а логические дефекты передаёт
парному `pravo-logika` флагом `[→ pravo-logika]`.
