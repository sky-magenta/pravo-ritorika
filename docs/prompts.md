# Шаблоны запросов

Готовые формулировки для каждого режима. Работают и как слэш-команды (плагин Claude Code), и как
обычный запрос (скилл подгрузится сам по `description`). Текст в `‹…›` замените своим.

Во всех режимах называйте **адресата** (суд, нотариус, госорган, оппонент, клиент) и **цель** —
без адресата риторический разбор не имеет точки отсчёта, и скилл его спросит.

## Стресс-тест — сильнейшая контрпозиция (диагностика)

```
/pravo-ritorika:stress-test ‹позиция; адресат и цель›
```
Обычным запросом:
```
Разбей мою позицию, сыграй оппонента: ‹текст›. Адресат — ‹суд / оппонент›.
Найди слабые места, куда ударит другая сторона: ‹текст›
Выдержит ли позиция сильнейшую контратаку: ‹текст›
```
Даёт: вердикт, позицию автора, steelman оппонента (его статус, удар по несущему, что он обойдёт
молчанием), карту уязвимостей (критичные / умеренные + способ укрепления), дефекты валидности
одной строкой `[→ pravo-logika]`, флаги к проверке. Позицию не переписывает.

## Статус — выбор линии позиции

```
/pravo-ritorika:status ‹ситуация или текст позиции; чего хотим добиться›
```
Обычным запросом:
```
На чём строить позицию — на факте, квалификации, оценке или процедуре: ‹ситуация›
Какую линию защиты выбрать: ‹ситуация›
Правильный ли фокус спора в этом тексте: ‹текст›
```
Даёт: ситуацию по четырём статусам (что доступно, что доказывать, чем платит позиция), фактический
статус текста и проверку на смешение линий, правило зеркала (статус оппонента), рекомендацию
условно по доказательственной базе. Решение о статусе оставляет автору.

## Структура — диспозиция текста

```
/pravo-ritorika:structure ‹меморандум / процессуальный документ / претензия / письмо›
```
Обычным запросом:
```
Проверь структуру аргументации: ‹текст›. Адресат — ‹клиент / суд›.
В каком порядке ставить аргументы: ‹перечень доводов›
Как выстроить меморандум для клиента: ‹текст›
```
Даёт: оценку, карту текста (части в фактическом порядке с ролью), таблицу структурных проблем,
предлагаемый порядок с обоснованием каждого перемещения. По явной просьбе — переписывает, меняя
порядок и связки, но не формулировки автора.

## Адаптация — переупаковка под адресата

```
/pravo-ritorika:adapt ‹позиция›  →  под адресата ‹суд / нотариус / госорган / оппонент / клиент›
```
Обычным запросом:
```
Переупакуй это для суда: ‹текст, изначально писавшийся для клиента›
Как подать это нотариусу / оппоненту в переговорах: ‹текст›
Адаптируй под переговоры, адресат — юрист контрагента: ‹текст›
```
Даёт: модель адресата (решение и рамки · опасения · язык · аргумент / шум), адаптированный текст
(отбор, порядок, регистр под адресата — содержание сохранено), список изменений с указанием
свойства адресата, проверку secondary-адресатов, флаги к проверке.

---

**Связка с `pravo-logika`.** Сначала валидность, потом убедительность. Рабочий цикл над позицией:
`/pravo-logika:review` → правка → `/pravo-ritorika:status` → `/pravo-ritorika:structure` →
`/pravo-ritorika:stress-test`. Логические дефекты, замеченные риторикой, помечаются `[→ pravo-logika]`
и отправляются в парный скилл; усиливать невалидный вывод скилл отказывается.

**Во всех режимах:** скилл усиливает **подачу** честной позиции, а не её содержание и не право по
существу. Нормы и практику по памяти не подставляет — ставит флаг `[требует проверки: норма / практика]`.
Факт принимает как данность автора. Каталог дефектов подачи — в [`cheatsheet.md`](cheatsheet.md),
сквозной пример — в [`example.md`](example.md).
